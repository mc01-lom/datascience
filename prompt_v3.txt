Crie um dashboard HTML standalone de Spend Analysis com as seções abaixo, na
ordem indicada. Use dados sintéticos representativos (~5.000 transações,
~38 fornecedores, 12 categorias N1, volumetria total ~R$10bi, período 2022–2026)
embutidos como array JSON diretamente no arquivo HTML, antes do encerramento da
tag </body>.

SCHEMA DE DADOS — campo por campo (uma transação = um objeto JSON)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Cada transação deve conter exatamente os seguintes campos:

  pedido               String   — identificador único da ordem de compra
  dt_pagamento         String   — "YYYY-MM-DD"
  ano                  Integer  — ex.: 2023
  mes                  Integer  — 1–12
  trimestre            String   — "Q1" | "Q2" | "Q3" | "Q4"
  valor_bruto          Float    — valor em R$ (ex.: 125000.00)
  quantidade           Integer  — unidades/ocorrências da transação
  nome_fornecedor      String   — razão social ou nome fantasia
  cnpj_raiz            String   — 8 dígitos (ex.: "12345678")
  categoria_n1         String   — um dos 12 segmentos UNSPSC do banco
  categoria_n2         String   — subcategoria dentro do N1
  esteira              String   — "Contratual" | "Pedido sem Contrato" |
                                  "Baixo Valor" | "Mercado IGA" |
                                  "Extra Compras" | "Fora da Política"
  flag_maverick        Boolean  — true se compra sem contrato vigente
  flag_tail_spend      Boolean  — true se fornecedor < R$100K/ano naquele ano
  flag_contrato        Boolean  — true se possui contrato ativo no período
  nome_area            String   — área requisitante (ex.: "Tecnologia")
  diretoria            String   — diretoria/superintendência do requisitante
  descricao_natureza   String   — natureza de pagamento (ex.: "Serviços de TI")
  status_contrato      String   — "Ativo" | "Vencido" | "Sem Contrato"
  dt_vencimento_contrato String — "YYYY-MM-DD" ou null se sem contrato
  flag_novo_fornecedor Boolean  — true na primeira transação do fornecedor no
                                  período 2022–2026

ATENÇÃO — coerência interna dos dados sintéticos:
  • flag_maverick = true ⟺ flag_contrato = false
  • esteira "Fora da Política" e "Pedido sem Contrato" devem ter flag_maverick = true
  • esteira "Contratual" deve ter flag_contrato = true e flag_maverick = false
  • dt_vencimento_contrato preenchida apenas quando flag_contrato = true
  • Incluir ~20 contratos com vencimento nos próximos 90 dias a partir de
    2026-06-01, e ~15 com vencimento entre 91–180 dias, para popular a seção 5.7
  • O salto de novos fornecedores em 2025 deve ser visível nos dados:
    flag_novo_fornecedor = true em ~800 transações de 2025 vs. ~150 em 2024
  • Distribuição de valor_bruto fortemente assimétrica (log-normal), com cauda
    longa: a maioria das transações entre R$1K–R$500K, alguns outliers > R$10M
  • 12 categorias N1 com pelo menos 3 subcategorias N2 cada; ao menos 3 categorias
    N2 com único fornecedor (para popular a seção 6.5 / 13)
  • Spend total dos 38 fornecedores somando ~R$10bi no período

Quatro cards dispostos em linha no topo da página:

  1. Spend Total (R$) — formatado como "R$ X,XXbi"
  2. Total de Transações — número inteiro com separador de milhar
  3. Total de Fornecedores Ativos — CNPJs únicos com pelo menos uma transação

Cada card deve exibir: ícone representativo, rótulo, valor principal.

SEÇÃO 8 — ÍNDICE DE CONCENTRAÇÃO HHI
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Pergunta de negócio: O spend está concentrado em poucos fornecedores?
Qual o risco de dependência por categoria?

Referência: U.S. DOJ — HHI < 1.000 = não concentrado |
            1.000–1.800 = moderadamente concentrado | > 1.800 = altamente concentrado

Fórmula: HHI = Σ (share_i)²  onde share_i = spend_i / spend_total × 100
(resultado entre 0 e 10.000)

8.1 Card HHI global
    Valor numérico do HHI calculado sobre todo o dataset
    Indicação visual da faixa com cor de fundo: verde/amarelo/vermelho
    Legenda das faixas DOJ visível no card

8.2 HHI por categoria N1
    Tipo: barras horizontais ordenadas por HHI decrescente
    Linhas de referência verticais: 1.000 (limiar moderado) e 1.800 (limiar
    alto)
    Colorir cada barra de acordo com a faixa que seu HHI pertence
    Tooltip: categoria, HHI, número de fornecedores, fornecedor dominante
    e share dele

SEÇÃO 10 — PRODUTO / SERVIÇO — NATUREZA DE PAGAMENTO
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Pergunta de negócio: O que o banco está comprando? A natureza do gasto é
consistente com a categoria? Há outliers por tipo de despesa?

Naturezas de pagamento a usar nos dados sintéticos:
  Serviços de TI · Consultoria Externa · Honorários Advocatícios ·
  Benefícios · Seguros · Facilities · Logística · Marketing · BPO ·
  Viagens · Material de Consumo · Impressão · Outros Serviços

10.1 Spend por natureza de pagamento
     Tipo: barras horizontais ordenadas por valor decrescente
     Exibir: valor absoluto (R$) e % do total

10.2 Heatmap natureza × categoria N1
     Tipo: heatmap (matriz)
     Linhas = naturezas de pagamento  |  Colunas = categorias N1
     Intensidade = spend no cruzamento
     Finalidade: identificar inconsistências (ex.: natureza "Benefícios" em
     categoria "Tecnologia")
     Tooltip: natureza, categoria, valor, % do total

10.3 Alerta de qualidade de dado — natureza genérica
     Tipo: card
     Métrica: % do spend classificado como "Outros Serviços"
     Cor: vermelho se > 10%, amarelo se 5–10%, verde se ≤ 5%

10.4 Box plot por natureza de pagamento (escala log10)
     Tipo: box plot
     Um box por natureza  |  Valores em log10(valor_bruto)
     Exibir: mediana, IQR (Q1–Q3), bigodes e outliers individuais
     Título deve indicar explicitamente: "Escala logarítmica (log₁₀) —
     aplicada para normalizar a assimetria da distribuição de valores"

10.5 Histograma geral de valores de transação (escala log10)
     Tipo: histograma
     Eixo X: log10(valor_bruto) com ticks rotulados em R$ (ex.: R$1K, R$10K,
     R$100K, R$1M, R$10M)
     Curva de distribuição normal sobreposta (linha tracejada)
     Título deve indicar: "Distribuição de valores — escala log₁₀"
     Anotações: indicar a moda, o valor mediano e o valor médio

REQUISITOS TÉCNICOS E DE DESIGN
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

ESTRUTURA DO ARQUIVO HTML:
  • Um único arquivo .html autocontido, sem dependências de servidor local
  • Plotly.js carregado via CDN (https://cdn.plot.ly/plotly-latest.min.js)
  • Array JSON dos dados embutido em uma tag <script id="rawData" type="application/json">
    antes do </body>; o código JavaScript deve ler via
    JSON.parse(document.getElementById('rawData').textContent)
  • Toda a lógica de cálculo (HHI, Pareto, ABC, YoY, etc.) implementada em
    JavaScript no próprio arquivo, executada sobre os dados JSON em memória

NAVEGAÇÃO:
  • Menu de navegação fixo no topo (sticky header) com âncoras para cada seção
  • Cada seção identificada por um <section id="secX"> correspondente
  • Largura máxima do conteúdo: 1400px centralizado

INTERATIVIDADE (Plotly nativa):
  • Todos os gráficos devem ter tooltips informativos:
    — Valores monetários no formato "R$ X,XXmi" ou "R$ X,XXbi"
    — Percentuais com uma casa decimal: "XX,X%"
    — Nome completo do fornecedor/categoria (sem truncamento no tooltip)
  • Gráficos com muitos itens (Pareto, tabelas longas): zoom e pan habilitados
  • Heatmaps: hover revela valores absolutos, não apenas intensidade
  • Gauge (5.1): sem interatividade necessária além do tooltip de valor

DESIGN E PALETA:
  • Cor de fundo do dashboard: branco (#FFFFFF) ou cinza muito claro (#F8F9FA)
  • Paleta corporativa:
    — Azul escuro: #1B3A6B (títulos, cabeçalhos, bordas de card)
    — Azul médio: #2E5FAC (barras principais, destaques)
  • Paleta de 12 cores para as categorias N1 — FIXA e CONSISTENTE em todo o
    dashboard (a mesma categoria N1 SEMPRE usa a mesma cor em qualquer gráfico):
    Definir um objeto JavaScript categoryColors = { "NomeN1": "#hexcode", ... }
    e referenciá-lo em TODOS os traces de todos os gráficos
  • Cores semânticas:
    — Verde: #28A745 (dentro da meta, positivo)
    — Amarelo/âmbar: #FFC107 (atenção, próximo do limite)
    — Vermelho: #DC3545 (fora da meta, alerta, risco)
  • Cada seção deve ter:
    — Título principal em azul escuro (#1B3A6B)
    — Subtítulo em itálico com a pergunta de negócio respondida pela seção
    — Benchmark/meta de referência exibido quando aplicável (ex.: "Meta Aberdeen:
      Contract Coverage > 80%")

FORMATAÇÃO DE VALORES:
  • Monetários: usar função formatBRL(value) que retorna:
    — "R$ X,XXbi" se value ≥ 1.000.000.000
    — "R$ X,XXmi" se value ≥ 1.000.000
    — "R$ X,XXK"  se value ≥ 1.000
    — "R$ X,XX"   se value < 1.000
    Usar vírgula como separador decimal e ponto como separador de milhar
    (padrão pt-BR)
  • Percentuais: sempre com uma casa decimal, ex.: "12,3%"
  • Escalas logarítmicas: sempre indicar no título ou subtítulo do gráfico
    que a escala é log₁₀ e o motivo ("para normalizar a assimetria da
    distribuição de valores")

PERFORMANCE:
  • O arquivo HTML completo (incluindo os ~5.000 registros JSON) deve carregar
    e renderizar todos os gráficos em menos de 5 segundos em um navegador moderno
  • Usar Plotly.newPlot (não addTraces iterativo) para cada gráfico
  • Calcular todos os agregados uma única vez no carregamento e armazenar em
    variáveis reutilizáveis — não recalcular por gráfico

COMPATIBILIDADE:
  • Deve abrir corretamente (sem erros no console) nos navegadores:
    Chrome 120+, Firefox 120+, Safari 17+, Edge 120+
  • Não usar APIs experimentais ou prefixos proprietários
  • Não requer conexão com internet além do CDN do Plotly.js no carregamento
    inicial
